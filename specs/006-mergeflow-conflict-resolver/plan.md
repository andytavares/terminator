# Implementation Plan: MergeFlow — Merge Conflict Resolver

**Input**: Feature specification from `specs/006-mergeflow-conflict-resolver/spec.md`

---

## Summary

MergeFlow adds an intent-first, card-based merge conflict resolution UI to the existing `git-integration` extension. It replaces the traditional three-panel diff view with a guided flow: Conflict Hub (file overview) → Conflict Resolver (one conflict at a time, with live preview and keyboard shortcuts) → Completion (stats + commit). The implementation is contained entirely within `extensions/git-integration/` using the existing git CLI subprocess pattern, Zustand for state, Zod for schema validation, and Vitest for tests. Deployment target is the **Terminator Electron desktop app** only — VS Code extension wrapper is out of scope.

---

**Storage**: electron-store for session persistence; git working tree for conflict file writes  
**Testing**: Vitest (same config as existing extension tests)  
**Target Platform**: Terminator Electron desktop app (macOS, Windows, Linux)  
**Project Type**: Desktop app extension panel  
**Performance Goals**: Conflict hub loads in < 2 seconds for repos with ≤ 50 conflicted files; conflict resolver renders each block in < 100ms  
**Constraints**: No new npm dependencies (Constitution IV); all code in `extensions/git-integration/`; 80% test coverage gate  
**Scale/Scope**: Supports merge sessions with up to 50 conflicted files, up to 20 conflicts per file

## Implementation Phases

### Phase A — Foundation (data layer + IPC)

1. **`merge-flow.schema.ts`** — Define all Zod schemas: `GitAuthor`, `ConflictBlock`, `ConflictFile`, `ConflictResolution`, `ResolutionDecision`, `ConflictSession`, `AISuggestion`. Export inferred TypeScript types.

2. **`conflict-reader.ts`** — Main-process module that:

   - Runs `git diff --name-only --diff-filter=U` to list conflicted files
   - For each file, reads `:1:`, `:2:`, `:3:` stages via `git show`
   - Parses working-tree file to identify conflict block boundaries and extract 4 lines of context
   - Detects `REBASE_HEAD` via `git rev-parse -q --verify REBASE_HEAD`
   - Reads author info from `git log --format="%an|%H|%ai" -1 HEAD -- <file>` and MERGE_HEAD equivalent
   - Returns `ConflictFile[]` ordered by conflict count

3. **`merge-flow.ipc.ts`** — Register all 8 IPC handlers (see contracts/ipc-channels.md). Each handler validates with Zod, delegates to `conflict-reader.ts` or `git-service.ts`, returns typed result or `{ error }`.

4. **`git.ipc.ts` modification** — Import and call `registerMergeFlowHandlers(register)` after existing handler registrations.

5. **`merge-flow.ts` (api/)** — Thin renderer wrapper over `extensionBridge.invoke` calls for all 8 channels.

**Tests** (written first):

- `conflict-reader.spec.ts`: mock `execFile`; test file parsing, block extraction, context lines, rebase detection, author extraction
- `merge-flow-ipc.spec.ts`: test each handler with valid + invalid payloads, error paths
- `merge-flow-schema.spec.ts`: Zod parse/reject for each schema

### Phase B — State management

6. **`merge-flow.store.ts`** — Zustand store implementing the full interface from `contracts/extension-api.md`. Includes:

   - Session lifecycle (start, clear)
   - Navigation (active file, active block, prev/next)
   - Decision confirmation + undo stack management
   - UI modal state (keepBoth, aiPanel)
   - Derived selectors (totalConflicts, totalResolved, isComplete, canUndo)

7. **`git.store.ts` modification** — Add `view: 'default' | 'merge-flow'` field and `setView()` action.

**Tests** (written first):

- `merge-flow.store.spec.ts`: test all store actions, undo stack behaviour, derived values, edge cases (undo with empty stack, complete session detection)

---

### Phase C — UI: Hub and entry point

8. **`GitSidebarPanel.tsx` modification** — Add "Resolve conflicts →" button rendered when `status.hasConflicts === true`. On click: sets `git.store.view = 'merge-flow'`.

9. **`GitFullView.tsx` modification** — Add `view === 'merge-flow'` branch that renders `<MergeFlowView repoRoot={repoRoot} />`.

10. **`MergeFlowView.tsx`** — Root component. On mount:

    - Calls `mergeFlowAPI.restoreSession(repoRoot)` — restores if a prior session exists
    - Otherwise calls `mergeFlowAPI.listConflicts(repoRoot)` and starts a new session
    - Renders `<ConflictHub>`, `<ConflictResolver>`, or `<CompletionScreen>` based on store state
    - Handles session-level errors via `useToastStore`

11. **`ConflictHub.tsx`** — File list view:
    - Summary bar: total files, total conflicts, estimated time
    - File list, sorted by conflict count (passed from store)
    - Per-file: file path, type badge, conflict count, both authors, complexity dot, "Resolved" check when done
    - Progress bar (totalResolved / totalConflicts)
    - Recommendation callout (first file by complexity)
    - "Commit merge" button (disabled until isComplete)

**Tests** (written first):

- `MergeFlowView.spec.tsx`: test session init branches (restore vs new), loading state, routing
- `ConflictHub.spec.tsx`: test file list rendering, sorting, progress bar %, commit button disabled state

### Phase D — UI: Conflict resolver (core interaction)

12. **`ConflictHeader.tsx`** — Progress dots (one per block: gray/blue/green), file breadcrumb, "← All files" back link, undo button (disabled when `canUndo === false`), conflict description text.

13. **`ConflictPanel.tsx`** — Two-column layout:

    - Left: "Your change" — author name, branch badge, commit hash, timestamp; diff lines (context dimmed to 50% opacity, added lines highlighted green)
    - Right: "Their change" — same structure, highlighted blue
    - No `<<<<<<<` markers rendered; panels show clean code

14. **`ResultPreviewStrip.tsx`** — Bottom strip, initially "Make a selection to preview the result." Updates to show merged code when `activeSelection` is set (on hover or after click). Expands with full preview once a selection is confirmed.

15. **`ActionBar.tsx`** — Five buttons with keyboard labels: Keep mine [M], Keep theirs [T], Keep both [B], Edit manually [E], Ask AI to suggest [Cmd+Shift+A]. On hover over first two: sets `previewSelection` in local state → `ResultPreviewStrip` reads it. On click: sets `activeSelection` in resolver state.

16. **`ConflictResolver.tsx`** — Composes the above. Manages:
    - Keyboard event listener (`M`/`T`/`B`/`E`/`Enter`/`←`/`→`/`Cmd+Z`/`Cmd+Shift+A`/`Esc`) scoped to this component, cleaned up on unmount
    - `previewSelection` (hover state, does not confirm)
    - `activeSelection` (clicked, not yet confirmed)
    - "Confirm & next →" button (appears once `activeSelection` is set)
    - On confirm: calls `mergeFlowAPI.resolveConflict()`, dispatches `confirmDecision()` to store, calls `persistSession()`, advances to next block
    - Shows `<KeepBothModal>` when `isKeepBothOpen` is true
    - Shows `<AiSuggestionPanel>` when `isAiPanelOpen` is true
    - Shows `<ManualEditor>` when edit-manually mode is active
    - `Cmd+Shift+A` opens AI panel; `Esc` closes AI panel or Keep Both modal (whichever is open)

**Tests** (written first):

- `ConflictHeader.spec.tsx`: dots, undo button disabled/enabled, back link
- `ConflictPanel.spec.tsx`: renders author info, applies correct CSS class for add/remove/context lines, no conflict markers
- `ResultPreviewStrip.spec.tsx`: initial placeholder, updates on preview prop change
- `ActionBar.spec.tsx`: hover triggers preview, click sets selection, Cmd+Shift+A label displayed
- `ConflictResolver.spec.tsx`: all keyboard shortcuts including Cmd+Shift+A, confirm flow, advance-to-next logic, Esc closes modals

### Phase E — UI: Additional screens

17. **`KeepBothModal.tsx`** — Modal overlay (not a full-screen route):

    - Two draggable code block cards with drag handles
    - "Mine first" / "Theirs first" toggle (swaps order without drag)
    - Live merged preview at bottom
    - Duplicate identifier warning (regex heuristic on method/function/class names)
    - Buttons: Cancel, "Let AI merge these" (disabled in this scope — shows `NOT_IMPLEMENTED` toast), "Use this order →"
    - On confirm: builds `resolvedText` from ordered blocks, closes modal, sets `activeSelection`

18. **`AiSuggestionPanel.tsx`** — Right-side panel (not replacing diff view):

    - Shows `AISuggestion` if available; shows loading state while fetch is in-flight
    - If `NOT_IMPLEMENTED`: shows "AI suggestions coming soon" placeholder with dismiss button
    - Buttons: "Accept suggestion," "Edit suggestion before accepting" (opens ManualEditor pre-populated with AI text), "Dismiss"
    - Background diff dims to 60% opacity while panel is open

19. **`ManualEditor.tsx`** — Replaces ConflictPanel when active:

    - Single `<textarea>` pre-populated with heuristic-better version (longer by line count; theirs wins on tie)
    - Syntax highlighting via `highlight.js` (already in `package.json`)
    - No autocomplete, no linting
    - "Save & next →" button

20. **`CompletionScreen.tsx`** — Full-panel view:
    - Success icon + "All conflicts resolved" headline
    - Time taken (startedAt → now) + total conflict count
    - Resolution summary stats: count per strategy (mine / theirs / both / ai / manual)
    - Per-file breakdown table (file path → strategy badges per block)
    - Editable commit message input (pre-filled from git's MERGE_MSG if present)
    - "Review changes" button (opens existing `FileDiffView` for each resolved file)
    - "Commit merge →" button:
      - Calls `mergeFlowAPI.mergeCommit()`
      - **On success**: calls `mergeFlowAPI.clearSession()`, shows success toast
      - **On failure** (HOOK_FAILED, TIMEOUT, NOTHING_TO_COMMIT, or any git error): shows error toast with the failure reason; stays on Completion screen so the user can retry or edit the commit message; session is NOT cleared

**Tests** (written first):

- `KeepBothModal.spec.tsx`: order toggle, preview update, duplicate warning trigger, confirm builds correct resolvedText
- `AiSuggestionPanel.spec.tsx`: NOT_IMPLEMENTED path renders placeholder, accept/edit/dismiss actions
- `ManualEditor.spec.tsx`: pre-population heuristic (longer wins, theirs on tie), save dispatches correct content
- `CompletionScreen.spec.tsx`: stats display, commit message editable, success path clears session + toast, failure path shows error toast + stays on screen

## Key Design Decisions (summary)

| Decision                   | Choice                                           | Rationale                                                              |
| -------------------------- | ------------------------------------------------ | ---------------------------------------------------------------------- |
| Deployment target          | Terminator Electron app only                     | VS Code extension wrapper is out of scope (confirmed in clarification) |
| Conflict data source       | `git` CLI subprocess                             | Consistent with git-service.ts; no new native deps                     |
| Session persistence        | electron-store (existing dep)                    | Atomic writes, crash-safe, zero overhead                               |
| AI integration scope       | Stub (`NOT_IMPLEMENTED`)                         | Phase 3 PRD; YAGNI per constitution                                    |
| Edit-manual pre-population | Longer block wins; theirs on tie                 | Documented heuristic from PRD                                          |
| Rebase label inversion     | Detect `REBASE_HEAD` at session start            | Covers both rebase modes; PRD requirement                              |
| Entry point                | `GitSidebarPanel` button → `GitFullView` routing | Same pattern as PR Review feature                                      |
| No new npm deps            | Zero additions                                   | Constitution Principle IV compliance                                   |
| Commit failure UX          | Error toast + stay on Completion screen          | Session preserved for retry; confirmed in clarification                |
| AI keyboard shortcut       | `Cmd+Shift+A`                                    | Chord prevents accidental activation; confirmed in clarification       |
